What This System Is

ASBL operates two AI products that talk to customers every day:

  • A text chatbot on the ASBL website — answers buyer queries about projects, prices, and availability.
  • A voice agent called Anandita — handles inbound and outbound phone calls from prospective buyers.

This QA system runs automatically every 4 hours. With zero human effort it:

  • Reads every chatbot conversation and phone call from the last 4 hours
  • Checks each one for errors — wrong prices, wrong RERA numbers, invented facts, missed leads, language failures
  • Validates whether business goals (site visits, OTP completions, affordability checks) are actually being met
  • Identifies every problem — single-agent bugs, cross-agent patterns, scoring miscalibrations
  • Tells the team exactly which file to open, what to change, and what will improve once the fix is made
⚡ Business Impact: The system catches pricing errors before they reach customers, flags wasted Meta ad spend in real time, and ensures Anandita always speaks the right language — without anyone reviewing calls manually.

Agent 1 — Chatbot QA
agents/chatbot_qa.py

Reads every chatbot conversation from the last 4 hours that has not yet been evaluated. Sends each one to Ollama along with the full knowledge base and asks it to find errors.

Step-by-Step Logic

1. Compute window: now minus 4 hours
2. Query prod MongoDB: conversations where createdAt >= window AND turnCount >= 2
3. For each conversation:
   a. Check qa_results.chatbot_qa — has this conversation_id been evaluated?
      YES → skip it (deduplication)
      NO  → continue
   b. Load agents/prompts/chatbot_qa.md from disk (fresh every time)
   c. Load all 8 KB files from knowledge_base/webbot/ (fresh every time)
   d. Strip HTML tags from bot responses
   e. Format as "User: ... \nBot: ..."
   f. Send to Ollama → receive JSON result
   g. Save result to qa_results.chatbot_qa
   h. If FAIL → print alert
4. Print: evaluated X conversations, skipped Y already done

Agent 2 — Voice QA (Anandita)
agents/voice_qa.py

Reads every phone call transcript from the last 4 hours. Checks Anandita's behaviour against her knowledge base and voice-specific rules.

Step-by-Step Logic

1. Compute window: now minus 4 hours
2. Query prod: calls where started_at >= window AND transcript is not empty
3. For each call:
   a. Check qa_results.voice_qa — has this call_sid been evaluated?

   b. Load agents/prompts/voice_qa.md (fresh from disk)
   c. Load 3 Anandita KB files: project_facts.md, system_prompt.md, qa_checklist.md
   d. Format transcript:
      speaker = "Anandita"     → "Anandita: ..."
      speaker = phone number   → "Caller: ..."
   e. Send to Ollama → receive JSON result
   f. Save to qa_results.voice_qa

Agent 3 — Analytics Validator
agents/analytics.py

Does not evaluate conversations. Validates whether the analytics scoring system — which assigns bands and fires Meta/Google conversion events — is actually producing the business outcomes you want. Runs 4 independent checks every cycle.

Check 1: Band Conversion Rates

Reads your business goals from config/outcomes.json. For each active outcome and each scoring band, checks whether the actual conversion rate matches your target.

For each active outcome (e.g. "site_visit_booked"):
  For each lead in scores_overall:
    band      = lead.lifetime.highest_band_ever
    converted = check outcome field (True / False)

  For each band:
    rate   = converted_count / total_count
    target = your target % from outcomes.json
    meta_events = count of events fired for this band

    FLAG (HIGH severity) if:
      meta_events > 0          ← money is being spent
      AND rate < target        ← conversions below target
      AND total_count >= 3     ← enough data to judge

Outcome Field Mapping (Internal — users never touch this)

Outcome NameMongoDB FieldMatch Value
site_visit_bookedmilestones.has_visit_bookedTrue (boolean)
otp_verifiedmilestones.has_otp_verifiedTrue (boolean)
affordability_confirmedmilestones.affordability_outcome"YES" (string)

Note: milestones.has_visit_confirmed was removed — this field is never populated in the ASBL system.

Check 2: Multiplier Effectiveness

Multipliers are score boosts triggered by specific user actions. This check verifies that each one actually correlates with better outcomes.

MultiplierWhat Triggers It
M1Affordability calculator (EMI-based, salary input) determines lead CAN afford
M2Lead stayed on site 30+ seconds after seeing they cannot afford it
M3Lead typed their actual home address (high intent signal)
M4Lead completed OTP verification — applies a 1.3× score multiplier
 
For each multiplier:
  Group A = leads who completed this multiplier
  Group B = leads who did NOT complete it

  rate_A = Group A leads who booked a visit / count(Group A)
  rate_B = Group B leads who booked a visit / count(Group B)

  effective = (rate_A > rate_B)

  FLAG (MEDIUM severity) if:
    NOT effective
    AND sample_size >= 5

Check 3: Meta Signal Accuracy

Meta/Google conversion events fire when a lead crosses a band threshold — each firing costs money. This checks what fraction actually convert to a site visit.

For each meta_conversion_event:
  Find the lead (via lead_id, or session_id → scores_session_wise)
  Check: has this lead booked a site visit?
    YES → led_to_visit += 1
    NO  → wasted += 1

accuracy = led_to_visit / total_events

FLAG (HIGH severity) if accuracy < 20% AND total_events >= 10
📊 Current finding: 0.38% accuracy — 52 out of 13,653 Meta events led to a confirmed site visit. ₹7,85,500 in Meta event value is being fired per Band5_Hot lead, but only ~1 in 263 converts.

Check 4: Funnel Gaps

GapQuerySeverity
leads_with_no_sessionscores_overall where lifetime.total_sessions == 0LOW
affordability_yes_no_visitscores_overall where affordability = YES AND has_visit_booked = FalseMEDIUM

Agent 4 — Feedback Agent (The Manager)
agents/feedback.py

The Feedback Agent is the manager. It reads all outputs from the QA agents, computes health scores, and asks the LLM to identify every genuine problem — including patterns that span multiple agents. It does NOT generate fixes.

🔑 Key design principle: The LLM sees everything and decides on its own what is wrong. There are zero rule-based checks. No hardcoded thresholds for problem detection.

Phase 1: Health Score Formula

Chatbot & Voice Formula

For each result in qa_results.chatbot_qa (last 4 hours): base_score = Ollama's raw score (1 to 10) confidence = Ollama's confidence (0.0 to 1.0, default 0.5 if missing) penalty = 0 for each issue: severity_weight = { HIGH: 2.0, MEDIUM: 0.8, LOW: 0.2 } penalty += severity_weight × confidence adjusted_score = clamp( base_score − penalty, min=0, max=10 ) health_score = average of all adjusted scores fail_rate = count(FAIL results) / total results

Why Confidence Matters

Example A: score=7, confidence=0.95, issues=[HIGH]
  penalty = 2.0 × 0.95 = 1.90  →  adjusted = 5.10  (very certain — hurts a lot)

Example B: score=7, confidence=0.40, issues=[HIGH]
  penalty = 2.0 × 0.40 = 0.80  →  adjusted = 6.20  (uncertain — hurts less)

A HIGH issue the LLM is unsure about should not tank the score the same
way a confirmed HIGH issue does.

Analytics Formula

high_flags = count of flags with severity = HIGH medium_flags = count of flags with severity = MEDIUM analytics_score = max(0, 10 − (high_flags × 2.5) − (medium_flags × 1.0))

Health Score Scale

Score RangeMeaning
8.0 – 10.0✅ Healthy — system performing well
6.5 – 7.9🟡 Acceptable — minor issues present
5.0 – 6.4🟠 Concerning — noticeable quality problems
3.0 – 4.9🔴 Bad — significant failures this cycle
0.0 – 2.9🚨 Critical — major breakdown, immediate action needed

Trend Detection

Look at the last 3 health scores in history: All three decreasing (each < previous) → DECLINING All three increasing (each > previous) → IMPROVING Otherwise → STABLE

Phase 2: Pure LLM Problem Detection

All data from this cycle is assembled and sent to Ollama. The LLM receives the full feedback_reasoning.md prompt (12KB) plus all QA results and analytics data.

Agent 5 — Recommendation Agent (The Engineer)
agents/recommendation.py

Receives the problem list directly from the Feedback Agent and produces one specific, actionable fix for each problem. Does NOT hunt for new problems.

Fix Generation Logic

For each problem (sorted: HIGH first, then MEDIUM, then LOW):

  Step 1: Check KNOWN_FIXES dictionary (no LLM needed — faster)
    DECIMAL_NUMBER     → template fix
    PRICE_ACCURACY     → template fix
    RERA_NUMBER        → template fix
    LANGUAGE_HANDLING  → template fix

  Step 2: If no template match → call Ollama with:
    [recommendation.md — 13KB: ASBL facts + file locations + fix patterns]
    [the problem: id, title, components, evidence, what_is_wrong]
    [historical baseline: first score, best score, current score, dates]

  Step 3: Return fix object

How Historical Baseline Is Used

For each component (chatbot, voice, analytics): Pull last 30 days of health_scores first_score = score on very first run + date best_score = highest score ever + date current = latest score + date LLM uses this to say: "Voice score was 7.5 on 2026-05-09 and dropped to 2.5 by 2026-05-12. Something likely changed around 2026-05-10."

Fix Object Schema

{
  "problem_id":       "P001",
  "problem_title":    "Anandita decimal format broken",
  "urgency":          "HIGH",
  "root_cause":       "One sentence including when it started",
  "fix":              "Exactly what to do — specific enough to act on today",
  "where":            "Exact file path, or 'scoring model', or 'CRM/ops workflow'",
  "change_type":      "kb_file_edit",
  "expected_outcome": "What improves, by how much, how quickly"
}

The Runner — How Everything Connects
runner.py
def run_once():
    chatbot_qa.batch(hours=4)       # Step 1
    voice_qa.batch(hours=4)         # Step 2
    analytics.run()                 # Step 3
    problems = feedback.aggregate() # Step 4 — returns Problem objects
    recommendation.run(problems)    # Step 5 — receives list directly

# runner.py once  →  run one full cycle right now
# runner.py       →  run every 4 hours indefinitely
💡 The problems list is passed directly in memory from Feedback → Recommendation. The Recommendation Agent does not re-fetch anything from the database.

The LLM Wrapper
llm.py
def ask(prompt) → str:
  Send prompt to http://localhost:11434
  Return raw text response

def _extract_json(text) → dict:
  1. Strip markdown code fences (```json ... ```)
  2. Brace-matching algorithm to find complete { ... } block
  3. Parse and return
  Raise ValueError if no valid JSON found

def ask_json(prompt, retries=2) → dict:
  Call ask(), then _extract_json()
  Retry up to 2 times on parse failure
  Raise on final failure

Ollama parameters:
  model: llama3.1:8b  |  stream: false
  temperature: 0.1    |  num_predict: 1500

How to Run the System

Prerequisites

  • Ollama app must be open (check Mac menu bar). Model llama3.1:8b must be pulled.
  • .env file must contain MONGO_URI_PROD and MONGO_URI_RESULTS.
# Install dependencies (one time)
pip3 install pymongo python-dotenv requests --break-system-packages

# Run one full cycle now (for testing)
cd /Users/aanirudhmehra/Desktop/asbl-qa-agent
python3 runner.py once

# Run every 4 hours (production)
python3 runner.py

# Individual agents
python3 agents/chatbot_qa.py           # last 4 hours of conversations
python3 agents/voice_qa.py             # last 4 hours of calls
python3 agents/analytics.py            # analytics checks
python3 agents/feedback.py aggregate   # health scores + problems
python3 agents/recommendation.py       # generates fixes

# Human feedback
python3 agents/feedback.py submit "Bot gave wrong Spectra price" "Anirudh"
python3 agents/feedback.py list 20
